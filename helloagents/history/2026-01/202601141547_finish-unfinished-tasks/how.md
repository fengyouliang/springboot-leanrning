# How: 执行策略与落地方案

## 总体策略

1. **先收敛入口**：补齐统一路线图页，并把路线图入口挂到 `helloagents/wiki/overview.md` 与各模块页面（`helloagents/wiki/modules/*.md`）。
2. **以“可跑证据链”驱动补齐**：优先补齐文档中 `对应测试：`、`入口：`、`Debug 建议` 等空块，统一指向方法级 `*LabTest` 入口。
3. **最小增量补 Lab**：当关键分支缺少可运行证据时，只新增最小 endpoint/配置/测试方法来钉死结论；避免大规模重构。
4. **导航不退化**：所有新增/改动文档保持“上一章/目录/下一章”与 README 索引可用，避免断链。
5. **分模块回归**：按模块执行测试，最后按需执行全仓回归。
6. **回填与审计**：把三份历史 `task.md` 的状态更新为可审计结果，并同步知识库与变更记录。

## 风险与规避

- 任务量大、跨模块：按“入口页 → 文档空块 → Lab 缺口”顺序分层推进，避免同时改动过多横切点。
- 测试回归风险：每个模块改完先跑该模块测试，避免到最后才发现问题。
- 文档一致性风险：统一使用相对链接；在关键索引页集中管理入口，减少漂移。

## 不做的事（若遇到则标记为 [-]）

- 与当前仓库结构/方向明显不一致的历史设想（例如已被后续实现替代）
- 引入新的第三方依赖或大版本升级
- 需要 CI/工作流改造才能闭环的“门禁类任务”（除非本次明确要求）
