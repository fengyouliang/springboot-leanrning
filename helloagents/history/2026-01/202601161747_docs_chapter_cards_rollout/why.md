# Why：全量文档书籍化改写（章节学习卡片：五问闭环）

## 背景

本仓库已经具备“多模块学习 + Book-only 主线之书 + Labs/Exercises 证据链”的雏形，但当章节数量变大后，读者会遇到两个常见问题：

1. **读完一章仍不确定“这一章到底在讲什么”**：知识点边界不清，容易把概念当成碎片记忆。
2. **知道结论，但不知道如何落地与验证**：缺少“怎么用 / 为什么 / 对应源码入口 / 可跑 Lab”的闭环，学习难以转化为排障能力。

你提出的目标非常明确：**每个章节都要说明**：

- 当前知识点是什么？
- 怎么使用？
- 原理是什么？
- 源码是什么（关键类/关键方法/断点入口）？
- 哪个 Lab 可以加深对当前知识点的理解？

## 范围（基于仓库现状的规模估算）

- 模块数量：18（`spring-core-*` + `springboot-*`）
- 模块章节（以各模块 `docs/README.md` 的链接清单为 SSOT）：174
- Book-only 主线之书页面：约 24（其中主线编号章 0–18 共 19）

> 说明：模块 docs 下的 Markdown 文件总数更多（包含 appendix/索引/目录页等），本计划以“章节清单 SSOT（docs/README.md 链接）”作为全量改写的主目标集合。

## 目标（Definition of Done）

对每个“章节”（SSOT 列表内）达成以下验收标准：

1. 章首提供**学习卡片**，完整回答“五问”（知识点/怎么用/原理/源码入口/Lab）
2. 明确给出**至少 1 个推荐 Lab**（可直接运行的命令或路径）
3. “源码入口”至少包含：
   - Spring/Spring Boot 关键类或关键方法（可作为断点锚点）
   - 本仓库对应的示例代码/测试入口（便于把机制落到可复现证据）
4. 通过仓库闸门：
   - `bash scripts/check-docs.sh`
   - `bash scripts/docs-site-build.sh`（`mkdocs build --strict`）

## 非目标

- 不引入新的技术栈（继续使用现有 MkDocs/Material/脚本体系）
- 不修改业务代码/测试逻辑（本计划聚焦“文档叙事与学习闭环”，不是新增功能）
- 不强制恢复旧的“章节契约 A–G”写法；采用更轻量、可渐进维护的“章节学习卡片”机制
